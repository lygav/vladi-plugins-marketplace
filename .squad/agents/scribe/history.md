**Purpose of Each File:**

1. **now.md**
   - What we're working on RIGHT NOW
   - Updated: At sprint planning, when priorities shift
   - Contains: version, focus_area, active_issues, agent roster

2. **wisdom.md**
   - Distilled patterns and principles
   - Updated: When discovering new patterns or correcting understanding
   - Contains: architecture decisions, conventions, anti-patterns, active patterns to preserve

3. **Agent History Files**
   - Role, responsibilities, key knowledge per agent
   - Updated: When agent learns new patterns
   - Contains: role-specific knowledge, SDLC rules, known issues

4. **ground-truth-v050.md**
   - Generated by code scan tool
   - Authoritative source: this is what code ACTUALLY does
   - When confused about implementation: read this, not READMEs
   - Triggers: docs audit, wisdom update, issue creation

### Documentation Standards

**Rules:**
1. **Docs live with code** — Update docs in same PR as code changes
2. **Ground truth from code** — Run scan if unsure what's implemented
3. **One source of truth per type:**
   - Code behavior: ground-truth-v050.md
   - Architecture decisions: wisdom.md
   - Roadmap: now.md
   - Implementation details: code comments + ground truth

### Ground Truth Scan Process

**When to run:**
- After major version release (like v0.5.0)
- When onboarding new team members
- When docs audit reveals significant discrepancies
- Quarterly or per sprint if high dev velocity

**What it generates:**
1. Architecture overview (layers, key abstractions)
2. SDK layer documentation (interfaces, schemas, base classes)
3. Placement abstraction (all implementations listed)
4. Communication abstraction (all implementations listed)
5. Registry system (enums, factories)
6. Knowledge management (LearningEntry structure)
7. Orchestration layer (agent coordination)
8. Configuration system (FederateConfig schema)
9. Script entry points (what each script does)
10. File formats & conventions (JSON, JSONL, YAML)

**Output:**
- `ground-truth-v050.md` — Complete, browsable reference
- Lines include code locations (file paths, line numbers)
- Real code snippets from actual implementation

### Knowledge Capture

**What to capture in wisdom.md:**
- Architectural decisions and rationale
- Conventions that work well
- Patterns that are generalizable
- Anti-patterns to avoid
- Rules for SDLC and code review

**What NOT to capture:**
- Temporary workarounds (track as issues instead)
- Personal opinions without evidence
- Things that should be in code comments
- Implementation details (those go in ground truth)

### Common Patterns to Document

When discovering patterns, add to wisdom.md:

**Good Patterns (preserve):**
- Zod schemas as single source of truth
- OTel emitter with no-op fallback
- Factory methods for extensibility
- Bootstrap idempotency
- Append-only learning logs

**Anti-Patterns (warn against):**
- Tight coupling between placement & communication
- Archetype-specific params in generic factories
- Docs that drift from code
- Backward compatibility promises pre-1.0
